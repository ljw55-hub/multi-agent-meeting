# 面试讲解稿

## 30 秒介绍

这是一个多 Agent 智能会议助手，用 FastAPI 做接入层，用 LangGraph 编排 5 个 Agent，完成从会议转写、纪要生成、待办提取、会议洞察到会后跟进的自动化流程。系统支持 REST 和 WebSocket 接入，并预留 PostgreSQL、Redis、ChromaDB，用于后续存储会议元数据、任务状态和历史会议语义检索。

## 1 分钟介绍

这个项目的核心是把会议处理拆成多个职责明确的 Agent：

- 转写 Agent 负责把音频转为带说话人和时间戳的文本。
- 纪要 Agent 负责把转写整理成议题、决策和下一步。
- 待办 Agent 负责提取负责人、任务和截止时间。
- 洞察 Agent 负责统计发言占比、关键词和会议效率。
- 跟进 Agent 负责汇总结果，生成会后报告，并预留外部系统同步。

编排上采用 Pipeline + Fan-out/Fan-in。转写必须先执行，纪要、待办、洞察可以并行，最后 Follow-up 汇总。

## 常见问题

### 为什么不用单个 Agent 做所有事？

单个 Agent 会导致 Prompt 复杂、输出不稳定、调试困难。拆成多个 Agent 后，每个 Agent 的输入输出更清晰，也方便单独替换模型、加缓存、做失败重试。

### 哪些逻辑用规则，哪些逻辑用 LLM？

确定性任务用规则，比如发言时长统计、发言占比、任务数量统计。语义理解任务用 LLM，比如纪要生成、待办抽取、洞察总结。

### 如果某个 Agent 失败怎么办？

每个 Agent 都只写自己的状态字段。失败时可以记录错误并返回 fallback 结果，不影响主流程继续向下执行。比如 LLM 不可用时，系统会用规则生成基础纪要和待办。

### 为什么用 LangGraph？

LangGraph 适合状态驱动的多 Agent 编排。它能把 State、Node、Edge 显式表达出来，比普通函数调用更容易解释、观测和扩展。

### ChromaDB 在项目中负责什么？

ChromaDB 用于存储历史会议内容的向量表示。后续可以支持“查询过去会议里关于某个项目的决策”和“基于历史上下文生成 follow-up”。

### Redis 在项目中负责什么？

Redis 可以存实时任务状态、WebSocket 会话状态、短期缓存和异步任务锁。当前 demo 先保留服务，后续接异步处理时会用到。

### PostgreSQL 在项目中负责什么？

PostgreSQL 适合存结构化数据，比如会议元数据、参会人、转写记录、待办事项、报告状态。

- 增加前端页面，展示会议报告和待办列表。
- 接入飞书/Jira，把待办同步到外部系统。
